# Fitness functions

**Project:** agent-builder
**Last updated:** 2026-07-12 (task 167: added F-015 runstore-isolation)

## What this file is

Fitness functions are **executable architectural invariants** — automated checks that verify the code still obeys the rules this project commits to. Layering, coupling, dependency direction, performance budgets, security thresholds, complexity limits.

This file is the **declarative spec** for those checks. The implementation lives in the runner the rules point to (a Makefile target, a tool config, a pytest file). This file does not describe how the checks are coded — it describes which invariants the code must satisfy.

## Why this is separate from the rest of the spec

Three things in this project enforce alignment between the code and what the spec claims. They have different jobs and run at different times:

| Mechanism | What it guards | When it runs |
|-----------|---------------|--------------|
| `spec-coverage-check` hook | Active task's TC markers must have test references before commit | Pre-commit (git commit) |
| `architect` drift-audit mode | Spec docs and diagrams still describe what the code does | On demand, periodically |
| **Fitness functions (this file)** | **Architectural invariants the code must always satisfy** | **Continuously — `make fitness` locally, also at Stop in `strict` profile** |

The drift-audit asks *"do the docs still describe the code?"* — semantic, agent-driven, episodic. Fitness functions ask *"does the code still obey the rules?"* — mechanical, executable, continuous. Both matter; neither replaces the other.

## How to run

```bash
make fitness          # run all fitness functions
make fitness-<rule>   # run one rule by name (see table below)
```

Add new rules by:
1. Append a row to the **Rules** table below
2. Add a `fitness-<rule>` target to the Makefile that runs the underlying tool
3. Add `fitness-<rule>` to the `fitness` umbrella target's prerequisites

If a rule starts failing intentionally (e.g. the rule was wrong, or the constraint has been deliberately relaxed), update or delete the row in the same commit as the relaxed code — don't leave a dead rule in the table.

For tool selection per language, see `references/fitness-functions.md` in the create-project skill.

## Rules

Keep entries concrete: the rule must be checkable by a tool, and the threshold must be a number or a yes/no, not a vibe. Delete rules that are no longer load-bearing. Each row should be earnable — write a one-line *why* in the row's description so a future reader (or future-you) can tell whether the rule is still load-bearing.

| F-006 | `internal/policy` is a leaf and `internal/runtime` reaches policy-engine only over IPC | structural | (1) `go list -deps ./internal/policy/...` reports no `agent-builder/internal/` path other than `internal/policy` itself; (2) `go list -deps ./internal/runtime/...` reports no `github.com/tkdtaylor/policy-engine` Go module import | 0 violations | `make fitness-policy-isolation` | block | ADR 038 mandates that `decide` runs out-of-process: policy-engine is a separate daemon reached over a Unix socket, never imported as a Go module. A `github.com/tkdtaylor/policy-engine` import in `internal/runtime` would expose an in-process `Decide()` the agent could self-grant through; a reverse import from `internal/policy` into other `internal/` packages would allow the agent's own code to influence the decision path. |
| F-007 | `internal/envelope` is a pure leaf and absent from `internal/supervisor`'s dependency graph | isolation | (1) `go list -deps ./internal/envelope/...` reports no `agent-builder/internal/` path other than `internal/envelope` itself (stdlib + `golang.org/x/crypto` only); (2) `go list -deps ./internal/supervisor/...` reports no `agent-builder/internal/envelope` import | 0 violations | `make fitness-envelope-isolation` | block | ADR 045 mandates that the cryptographic primitives (`internal/envelope`: Ed25519, X25519+AEAD, replay cache) stay on the channel side of the `supervisor.GoalSource` seam. The envelope carries secrets (private keys, nonces) and crypto implementation details; it must never enter the dumb supervisor's import graph. The supervisor stays a host-side control core with no transport/crypto in its transitive imports (F-003 refinement). |
| F-008 | Default run pipeline wires the exec-sandbox block backend | structural | `go list -deps ./internal/runtime/...` reports an `internal/sandbox/execsandbox` import in the default run path | import present | `make fitness-exec-sandbox-default` | block | ADR 035 swapped the default run backend to the shipped `exec-sandbox` block (`internal/sandbox/execsandbox`), completing the bootstrap's self-leveraging swap. If `internal/runtime` stopped wiring execsandbox, the agent would silently fall back to the older Podman adapter and the "runs on the block it built" property would regress unnoticed. |
| F-009 | Mermaid diagrams render on GitHub | hygiene | Scan of `README.md` and every `*.md` under `docs/` reports zero GitHub-render hazards (syntax GitHub's Mermaid parser rejects, e.g. `;` in labels or `[]` in sequence messages) | 0 render hazards | `make fitness-diagrams-render` | block | `docs/architecture/diagrams.md` is authoritative spec; a Mermaid block with a GitHub parse error renders invisible to a reader, silently drifting from the code without notice. F-009 makes "the diagrams render" an executable invariant checked at every gate run. |
| F-010 | `internal/orchestrator` has no DIRECT import of `internal/executor` | structural | `go list -f '{{ .Imports }}' ./internal/orchestrator` (the package's own direct imports, NOT `-deps`) reports no `github.com/tkdtaylor/agent-builder/internal/executor` path | 0 violations | `make fitness-orchestrator-no-executor` | block | ADR 042/046: the Tier-1 orchestrator authors no code and must never reference the executor directly. It dispatches workers THROUGH `internal/runtime`, which legitimately imports `internal/executor` — that transitive path is the blessed dispatch path (the worker runs the executor inside its box). So the invariant is a **direct**-import assertion (a `-deps` transitive check would false-positive on the legitimate runtime path). A direct executor import in the orchestrator would mean the coordination layer itself could author/run code, collapsing the no-code-authoring bright line. |
| F-011 | `internal/channel/worker` (orchestrator↔worker transport) is a leaf with only envelope/supervisor/audit DIRECT internal imports | isolation | `go list -f '{{ .Imports }}' ./internal/channel/worker` (the package's own direct imports, NOT `-deps`) reports no `github.com/tkdtaylor/agent-builder/internal/` path other than `internal/envelope`, `internal/supervisor`, and `internal/audit` | 0 violations | `make fitness-worker-transport-isolation` | block | ADR 048 §3: the orchestrator↔worker transport adapter signs/seals/replay-checks work-items and results at the trust boundary (ADR 042). It must stay a leaf so the crypto/transport stays off every other package's graph: its only collaborators are `internal/envelope` (the sign/verify/seal/replay primitive), `internal/supervisor` (the `Task`/`Result` seam types), and `internal/audit` (the rejection-event `Sink`, itself a verified leaf via F-005). This is a **direct**-import assertion (like F-010): `internal/supervisor` legitimately drags in `internal/gate`/`internal/sandbox` transitively, so a `-deps` check would false-positive on that blessed path; the security intent (keep `internal/executor`/`internal/runtime`/`internal/orchestrator`/LLM/web-fetch off the transport) is fully covered by the direct check because none of the three allowed leaves imports those (F-003/F-005/F-007), so the transport can only reach them via a direct import this check blocks. |
| F-012 | `internal/memoryguard` is a leaf: its transitive dependency graph contains no other `agent-builder/internal/` package | isolation | `go list -deps ./internal/memoryguard/...` reports no `github.com/tkdtaylor/agent-builder/internal/` path other than `internal/memoryguard` itself (only stdlib imports) | 0 violations | `make fitness-memoryguard-isolation` | block | ADR 049: the memory-guard IPC adapter must stay a portable leaf so it can be wired into the orchestrator (or any future caller) without introducing layering cycles. If `internal/memoryguard` imported `internal/orchestrator`, `internal/runtime`, or any other agent-builder internal package, a caller importing `internal/memoryguard` would transitively acquire those dependencies — violating the block-adapter discipline established in ADR 026/038. Keeping it stdlib-only mirrors `internal/policy`, `internal/vault`, and `internal/audit` (F-005/F-006). |
| F-013 | No registered recipe targets the agent-builder own-repo as a result sink | security | `grep` of recipe source (`internal/recipe`, excluding `_test.go` and import lines) reports no line where `github.com/tkdtaylor/agent-builder` co-occurs with a `sink`/`remote`/`publish` token | 0 violations | `make fitness-no-self-repo-sink` | block | ADR 042's non-negotiable bright line is "no agent at any tier edits agent-builder's own repo." Task 085 enforces it two independent ways (ADR 050 §2): a RUNTIME deny (the orchestrator's `spawn-worker` decision refuses any worker whose `target_repo`/`sink` is the own-repo, fail-closed via `decideSpawnWorker`/`targetsOwnRepo`) and this STATIC check (a recipe that declared the own-repo as a sink could not even register without the gate firing). Either alone is a single point of failure; both make the line unreachable by construction and at runtime. The detection predicate is unit-tested (`orchestrator.SelfRepoSinkViolation` + TC-085-05 fixture run asserting a non-zero exit on a violation fixture and exit 0 on a clean one). |
| F-015 | `internal/runstore` (the durable run journal) is a leaf: its transitive dependency graph contains no other `agent-builder/internal/` package | isolation | `go list -deps ./internal/runstore/...` reports no `github.com/tkdtaylor/agent-builder/internal/` path other than `internal/runstore` itself (only stdlib imports) | 0 violations | `make fitness-runstore-isolation` | block | ADR 065: the file-backed run journal must stay a portable, stdlib-only leaf so the orchestrator (task 168) or any future caller can compose it without introducing a layering cycle. If `internal/runstore` imported `internal/orchestrator`, `internal/runtime`, or any other agent-builder internal package, a caller would transitively acquire those dependencies, violating the same block-adapter discipline F-012 enforces for `internal/memoryguard` and F-005/F-006 for `internal/audit`/`internal/policy`. Keeping it stdlib-only is what makes the journal safe to wire into the trusted core later. |
| F-014 | `internal/orchestrator/planner` (the LLM-backed `Planner` concrete) has no DIRECT import of `internal/executor` | structural | `go list -f '{{ .Imports }}' ./internal/orchestrator/planner` (the package's own direct imports, NOT `-deps`) reports no `github.com/tkdtaylor/agent-builder/internal/executor` path | 0 violations | `make fitness-llm-planner-no-executor` | block | ADR 046 §6: the LLM-backed `LLMPlanner` decomposes a goal by routing a prompt through a model, but — like the orchestrator under F-010 — it must NOT acquire an unrestricted model path by importing `internal/executor` directly. It reaches the model through the router/registry path (`internal/router`, which itself imports `internal/executor`) via the narrow `ExecutorResolver`/`Invoker` seams the planner package defines; the `*router.Router` satisfies `ExecutorResolver` only at the wiring layer (`internal/cli`), where the executor import already lives. So the invariant is a **direct**-import assertion (a `-deps` transitive check would false-positive on the legitimate router path), identical in form and rationale to F-010. A direct executor import in the planner would let the decomposition step author/run code outside the dispatch path, collapsing the no-code-authoring bright line one layer down. |

Categories: `structural` (cycles, layering, dependency direction), `hygiene` (logging, leftovers, debug code), `performance` (latency, throughput, memory), `complexity` (cyclomatic, file size, fan-out), `security` (deps, surface, secrets), `coverage` (test coverage thresholds).

Severity:
- `block` — fitness check exits non-zero; the runner reports a failure. Fix the violation or relax the rule deliberately.
- `warn` — surfaces in output but does not fail the runner. Use for budgets that may have a temporary justified excursion.

## Notes

- Rules in this file are the *project's* commitments, not generic best practices. Don't add a rule because it's a popular metric — add it because violating it would break something this project promises.
- Fitness functions should fail fast and have low false-positive rates. A rule that flags 50 things every run gets ignored.
- The hook only runs at `strict` profile so fast iteration isn't slowed down. Run `make fitness` manually before any milestone where invariants matter.
